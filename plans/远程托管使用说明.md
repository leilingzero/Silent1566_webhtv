# WebHTV 远程托管使用说明

## 1. 这个功能解决什么问题

远程托管的目标是让一台 WebHTV 设备可以被另一端远程管理，例如：

- 远程查看设备在线状态。
- 远程推送播放地址。
- 远程发起搜索。
- 拉取最近调试日志。
- 后续扩展远程同步、配置管理、文件管理等能力。

当前版本先落地 App 被控端基础能力：App 可以注册到中转服务、生成绑定码、保持在线轮询、接收基础命令。完整 Web 控制台和更完整的 App 主控端还属于下一阶段。

## 2. 先理解三个概念

- 中转服务 URL：你自己部署的远程 relay 地址，例如 `https://xxx.workers.dev`、`https://xxx.deno.dev`、`https://xxx.vercel.app`。
- 被控端：需要被远程管理的 WebHTV App，通常是电视、盒子或长期在线设备。
- 主控端：负责输入绑定码、保存设备组令牌、下发远程命令的一端。可以是 Web 控制台，也可以是未来另一台 WebHTV App。

简易中转服务不保存长期账号和绑定关系。长期令牌保存在 App/主控端本地。只要中转服务域名不变，服务重启后通常不需要重新绑定。

## 3. 部署中转服务

仓库里目前提供三种零配置部署方式，选一种即可。

### 3.1 Cloudflare Worker

适合长期使用，推荐优先选择。

```bash
cd serverless/webhtv-remote-simple
npm install
cp wrangler.toml.example wrangler.toml
npm run deploy
```

部署完成后会得到类似：

```text
https://webhtv-remote-simple.xxx.workers.dev
```

这个地址就是 App 里要填写的“中转服务 URL”。

### 3.2 Deno Deploy

在 Deno Deploy 新建项目，入口选择：

```text
serverless/webhtv-remote-deno/main.js
```

部署完成后会得到类似：

```text
https://xxx.deno.dev
```

### 3.3 Vercel Edge

在 Vercel 新建项目时，把项目根目录设为：

```text
serverless/webhtv-remote-vercel
```

入口文件是：

```text
api/index.js
```

部署完成后会得到类似：

```text
https://xxx.vercel.app
```

## 4. 验证中转服务是否可用

浏览器打开：

```text
https://你的中转服务域名/api/server/capabilities
```

能看到类似下面的 JSON，就表示中转服务可用：

```json
{
  "ok": true,
  "serverMode": "simple",
  "relayMode": "origin-token-memory"
}
```

如果打不开，先检查部署平台的域名、路由和网络，不要先排查 App。

## 5. 在被控端 App 注册

在需要被远程管理的 WebHTV 设备上操作：

1. 打开 `设置 -> 增强功能 -> 远程托管`。
2. 在“中转服务 URL”里填写部署后的域名，例如：

```text
https://xxx.workers.dev
```

3. 保持“启用此服务”勾选。
4. 如果希望电视长期在线接收命令，勾选“保持远程托管在线”。
5. 点击“保存/注册”。

注册成功后，本机会生成并保存 `deviceToken`。后续 App 启动会自动向同一个中转服务恢复在线状态。

## 6. 生成绑定码

被控端注册后：

1. 点击“生成绑定码”。
2. App 会显示 6 位绑定码。
3. 点击绑定码可以复制。
4. 绑定码短时间有效，过期后重新生成即可。

绑定码不是长期密码。它只是一次性把这台设备加入某个设备组的入口。

## 7. 主控端如何绑定设备

当前 App 端已经能生成绑定码和接收入组命令，但完整图形化 Web 控制台还不是本次实现内容。主控端需要调用中转服务的：

```text
POST /api/groups/claim
```

请求体最少包含：

```json
{
  "code": "123456",
  "alias": "客厅电视"
}
```

如果是把设备加入已有设备组，还需要带已有 `groupToken`：

```json
{
  "code": "123456",
  "groupToken": "gtk_xxx",
  "alias": "客厅电视"
}
```

服务端返回 `groupId/groupToken/groupTokenHash/deviceId` 后，主控端要把 `groupToken` 保存在本地。服务端会给被控端排队一个内部命令 `remote.profile.addGroup`，被控端轮询到后会写入设备组。

绑定完成后，被控端界面里的状态会从“未绑定”变成包含服务数和设备组数量的状态。

## 8. 界面按钮说明

- 中转服务 URL：填写你部署的 relay 域名。只认域名 origin，路径、参数不会作为绑定边界。
- 启用此服务：允许 App 使用这个中转服务注册和轮询。
- 保持远程托管在线：启动前台服务和保活锁，更适合电视、盒子等长期在线设备。
- 保存/注册：保存 URL，生成或复用本机设备令牌，并调用 `/api/device/register`。
- 生成绑定码：生成短期绑定码，供主控端调用 `/api/groups/claim`。
- 打开控制台：打开中转服务域名。当前简易中转服务默认主要是 API，不一定有图形化控制台页面。
- 清空托管：删除本机保存的远程托管 profile、设备令牌、设备组和待绑定记录。

## 9. 当前版本支持的远程命令

当前 App 被控端已经支持：

- `device.status`：返回设备、App 版本、设备组数量、调试日志状态。
- `action.search`：在 App 内触发搜索。
- `action.push`：向 App 推送播放地址。
- `log.recent` / `device.log.recent`：返回最近调试日志。
- `remote.profile.addGroup`：绑定过程中由服务端下发，用于把被控端加入设备组。

还没有完整实现：

- App 内主控端图形界面。
- 完整 Web 控制台。
- 配置列表、配置切换、配置删除。
- 远程一键同步导出/恢复。
- 远程文件管理。
- Git 云盘和远程托管的联动。

## 10. 数据保存和安全注意事项

App 会把远程托管资料保存到：

```text
WebHTV/RemoteTrust/profile.json
```

同时会在内部偏好里保存一份镜像。建议授予文件权限，这样卸载重装后仍可恢复托管关系。

普通一键同步和备份会排除：

```text
remote_trust_*
WebHTV/RemoteTrust/
```

这样可以避免把一台设备的 `deviceToken/groupToken` 同步到另一台设备，造成设备身份被克隆。

不要把 `profile.json`、`deviceToken`、`groupToken` 发给别人。拿到这些令牌的人可以控制对应设备组。

## 11. 常见问题

### 11.1 为什么填了 URL 后提示无效

需要填写完整可访问域名，例如：

```text
https://xxx.workers.dev
```

只填空字符串、错误协议或不可解析地址都会失败。

### 11.2 为什么“打开控制台”没有页面

当前简易中转服务主要提供 API，不保证有图形化首页。“打开控制台”只是打开中转服务域名。后续如果增加 Web 控制台，这个按钮会直接进入控制台页面。

### 11.3 绑定码生成了，但设备没有加入组

绑定码只是第一步。还需要主控端调用 `/api/groups/claim`。如果中转服务实例在 claim 后、被控端轮询前被平台回收，内部入组命令可能丢失，需要重新生成绑定码或由主控端补发 `remote.profile.addGroup`。

### 11.4 中转服务重启后要重新绑定吗

同一个域名不变时，一般不需要。App 会用本地保存的 `deviceToken/groupToken` 重新注册并恢复在线路由。

### 11.5 什么时候勾选“保持远程托管在线”

电视、盒子、常年供电设备建议勾选。手机、平板如果担心耗电，可以不勾选，只在 App 进程存活时接收远程命令。

## 12. 当前推荐使用流程

1. 部署 Cloudflare Worker 简易中转服务。
2. 浏览器访问 `/api/server/capabilities`，确认可用。
3. 在电视 WebHTV 打开 `设置 -> 增强功能 -> 远程托管`。
4. 填写中转服务 URL。
5. 勾选“启用此服务”。
6. 电视长期在线时勾选“保持远程托管在线”。
7. 点击“保存/注册”。
8. 点击“生成绑定码”。
9. 在主控端输入绑定码并调用 `/api/groups/claim`。
10. 等待电视端轮询到入组命令，状态显示设备组数量后，再开始远程命令测试。
